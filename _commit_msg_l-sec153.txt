L-sec153: /admin 루트의 레거시 마스터 패스워드 로그인 UI 제거

문제 (실제 발견된 보안 공백)
  사용자가 wishes.co.kr/admin 에 접속하면 단일 비밀번호 한 줄짜리
  "관리자 로그인 / 암호를 입력하세요" 화면이 그대로 노출되어 있었음.
  이 화면은 src/app/admin/page.tsx 가 서비스하는 것으로,
    handleLogin() → /api/admin/stats 에 Bearer 로 password 전송
    → verifyAdminAuth 의 MASTER_PASSWORD (env) 경로로 통과
    → sessionStorage('admin_password') 에 저장 후 전권 대시보드 진입
  이었음. admin-auth.html 에서 구축한 Supabase Auth + admin_users
  role/status 체계(L-sec97/L-sec127/L-sec48 등)를 전부 우회하는
  "쪽문" 으로, 마스터 패스워드 하나만 유출되면 다음이 가능:
    - /api/admin/* 전 엔드포인트 접근 (매물/상담/통계/로그)
    - admin-auth.html 경로로 가입 승인 대기 중인 모든 계정 승인/차단
  보안 감사 L-sec27/113/136 에서 /search 쪽 admin_password 노출은
  차단했지만, 정작 일반 진입 URL /admin 은 구 화면 그대로 방치돼 있었음.

조치
  src/app/admin/page.tsx:
    1) useState 'authError' 제거. useState 'password' 는 내부 Bearer
       토큰 홀더로만 유지 (대시보드 호출에 쓰임).
    2) useEffect 토큰 해석 우선순위 변경:
         1순위: sessionStorage 'ws_token'        — admin-auth.html Supabase JWT
         2순위: sessionStorage 'admin_password'  — 레거시 fallback (Phase 3c 제거 예정)
         3순위: localStorage 'admin_password' → sessionStorage 이관
       아무것도 없으면 window.location.replace('/admin/admin-auth.html').
    3) handleLogin 제거. 로그인 입력 자체를 프런트에서 받지 않음.
    4) isAuthenticated=false 일 때 렌더되던 <form> 전체 제거 →
       "로그인 페이지로 이동 중..." 안내 + /admin/admin-auth.html 링크만.

효과
  - 레거시 마스터 패스워드를 프런트에서 입력받을 수 있는 UI 경로 소멸.
  - /admin 방문 시 Supabase 이메일/비밀번호 + role 체계로 반드시 진입.
  - 기존에 sessionStorage 에 ws_token 이 있는 세션(슈퍼어드민 포함)은
    리다이렉트 없이 기존 대시보드 즉시 진입 (회귀 없음).

남은 숙제 (의도적으로 뒤로 분리)
  서버측 verifyAdminAuth 는 아직 MASTER_PASSWORD(env) 토큰을 accept.
  이유: 크롤러 브리지(WISHES_CRAWLER_BRIDGE_TOKEN) + 백엔드 내부 호출
  스크립트가 동일 env 에 의존. 갑작스런 제거는 회귀 위험.
  후속(L-sec154 / Phase 3c)에서:
    - MASTER_PASSWORD 경로 삭제
    - 크롤러 브리지는 dedicated env 토큰만 허용
    - sessionStorage 'admin_password' 레거시 key 완전 제거
    - ws_token 을 HttpOnly ws_session 쿠키로 전환 (phase 3 로드맵)

검증
  - TypeScript: authError/setAuthError/handleLogin 참조 없음 (grep 결과 0건).
  - 렌더 분기: 미인증 시 폼 노출 없음, 링크 1개만 표시.
  - 데이터 호출 경로(fetchData) 는 ws_token 기반으로 그대로 작동.
