### C-1. 관리자 IDOR: DELETE/PATCH `/api/admin/listings/[id]` 소유권 미검증

**증거** (`src/app/api/admin/listings/[id]/route.ts` DELETE 핸들러, 확인됨):
```ts
export async function DELETE(request, { params }) {
  if (!(await verifyAuth(request))) return 401;
  const { id } = await params;
  // ❌ listing.created_by vs 현재 사용자 UID 비교 없음
  await supabase.from('listings').delete().eq('id', listingId);
}
```

**영향도**: 중대. WISHES 는 앞으로 다중 에이전트 운영(에이전트 A 가 자기 매물만 등록/관리)을 전제로 L-v7 `scope=mine` 을 구축했다. 그런데 편집/삭제 API 는 여전히 "관리자 이면 전부 가능" 상태. 에이전트 B 가 에이전트 A 의 매물 ID 를 찍어 DELETE 호출 → 복구 불가.

**수정안** (L-sec112 권장):
```ts
// 1) DELETE 전에 소유권 조회
const { data: row } = await supabase
  .from('listings')
  .select('id, created_by')
  .eq('id', listingId).single();

// 2) superadmin 이거나 본인 매물일 때만 허용
const uid = await extractUidFromRequest(request);
const role = await getRoleByUid(uid);
if (row.created_by && row.created_by !== uid && role !== 'superadmin') {
  return 403;
}
```

**함께 고칠 곳**:
- `[id]/route.ts` PATCH (상태 변경)
- `[id]/route.ts` PUT (편집)
- `/api/admin/listings/route.ts` PUT (편집 전용 경로가 따로 있는지 확인)

- `/api/admin/listings-bulk-update/route.ts` (존재 시 동일)

**소프트 삭제 전환(권장)**: 즉시 hard delete 대신 `status='deleted' + deleted_at + deleted_by` 로 전환. 7일 후 cron 으로 hard delete. 에이전트 실수/의도적 파괴 방어.

### C-2. `admin_password` 를 sessionStorage 에 평문 저장

**증거** (grep 결과로 확인됨):
```
src/app/admin/page.tsx:174  sessionStorage.setItem('admin_password', password);
src/app/search/page.tsx:24  sessionStorage.setItem('admin_password', p);   // 마이그레이션
src/app/admin/page.tsx:148  savedPassword = sessionStorage.getItem('admin_password');
```

L-sec54 로 localStorage → sessionStorage 로 옮기면서 리스크를 줄였지만, **어떤 XSS 라도 sessionStorage 를 읽을 수 있다.** 동일 오리진의 공개 경로(`/listings/[id]`, `/search`, `/map`, `/board`) 어느 한 곳에서 `innerHTML` 기반 미escape 렌더링이 한 번이라도 일어나면 관리자 비밀번호가 즉시 유출된다.

**영향도**: 매우 심각. 한 번의 XSS 로 관리자 세션이 아니라 **비밀번호 자체**가 유출됨. 비밀번호는 리셋하지 않는 한 영구 유효.

**2026 기준 개선안** (L-sec113~115 묶음):
1. **아키텍처**: `admin_password` 기반 bootstrap 제거. Supabase `supabase.auth.signInWithPassword()` + 서버 발급 Bearer JWT(현재 사용 중) **만** 신뢰.
2. **저장**: JWT 는 **HttpOnly, Secure, SameSite=Strict cookie** 로 이동. sessionStorage 사용 중단. 클라이언트에서 `document.cookie` 로도 읽을 수 없게 함.
3. **만료**: Access token 15분, refresh token 7일, refresh는 rotating(one-time use).
4. **전환 전략**: 
   - Step 1: 신규 로그인 플로우 구현(쿠키 기반), 기존 sessionStorage 병렬 허용
   - Step 2: 모든 관리자 계정 강제 재로그인 공지 → 2주 후 localStorage/sessionStorage 코드 제거
   - Step 3: L-sec54 코드(`admin_password` 관련 읽기/쓰기 전부) 완전 삭제

**즉시 미티게이션 (L-sec113 hotfix)**: CSP 에 `script-src 'self' 'nonce-*'` 강화(현재 `unsafe-inline` 있으면 제거), Trusted Types 도입, DOMPurify 로 모든 사용자 입력 렌더링 wrap.

### C-3. L-sec106 불완전 — `/api/ai/match` PostgREST injection 미차단

**증거** (`src/app/api/ai/match/route.ts:65,75`, 확인됨):
```ts
if (filters.dong) q = q.ilike('dong', `%${filters.dong}%`);
if (filters.businessType) q = q.ilike('business_type', `%${filters.businessType}%`);
```

L-sec106 에서 `/api/listings/route.ts` 와 `/api/map/search/route.ts` 의 `.or()` 에는 `qEscaped.replace(/"/g, '""').replace(/%/g, '\\%')` 를 적용했지만, **같은 파일/다른 호출 경로에서 ilike 는 escape 없이 템플릿 리터럴로 붙인다.** `filters` 는 AI 가 사용자 쿼리를 구조화해 내뱉은 JSON 이라 공격 면적은 간접적이지만, AI 프롬프트 주입으로 `filters.dong = '%" OR status."eq."공개%'` 같은 값이 흘러들 경우 predicate 무력화가 이론상 가능.

**수정안** (L-sec112 와 함께 L-sec113 로 묶음):
```ts
const escapeIlike = (s: string) => s.replace(/[%_\\]/g, '\\$&');  // PostgREST ilike wildcards 차단
if (filters.dong) q = q.ilike('dong', `%${escapeIlike(filters.dong)}%`);
```

**함께 검사할 패턴** (`grep -rn "\.ilike\|\.like" src/`):
- 모든 `.ilike()` / `.like()` 호출부에 사용자 입력이 있으면 escape 적용
- `.or()`, `.filter()`, `.match()`, `.textSearch()`, `.rpc()` 호출도 1차 스캔

### C-4. scope=mine 캐시 키 폴백 취약

**증거** (`src/app/api/admin/listings/route.ts:~182-184, 270-273`):
- `scope === 'mine'` 이고 `scopeUid` 추출 성공 시: cacheKey=`['listings-minimal-v3-mine', scopeUid]` → 사용자별 분리
- `scopeUid === null` (인증 헤더는 있지만 JWT 파싱 실패) 시: **빈 배열 응답 + 캐시 미사용** 으로 이미 처리됨(L-v7 확인)

에이전트 2 의 우려(여러 사용자가 null 키로 섞임) 는 현재 코드로는 트리거되지 않는다. **단** 향후 누가 이 경로에 캐시를 추가하면 리그레션 가능 → 주석에 경고 + 단위 테스트 추가 필요.

**소프트 개선안**: `if (!scopeUid && scope === 'mine') return 401;` 을 명시적으로 하고 fallback 경로 제거. 현재처럼 빈 배열 + `scope_auth: 'failed'` 도 OK 지만 401 이 더 명확.

## 6. 혁신 제안 — 2026 tier "best of best"

WISHES 가 국내 부동산 플랫폼 중 선두로 올라서려면 단순 결함 수정 이상이 필요하다. 아래는 **기술적 차별화**가 가능한 방향들이다.

### 6-1. Zero-Trust 관리자 아키텍처

- **Passkey/WebAuthn 필수화**: 관리자는 기기 기반 passkey 만 허용. 비밀번호 제거.
- **Just-In-Time Elevation**: 위험 작업(대량 삭제, 사용자 권한 변경)은 요청 순간 재인증(WebAuthn 터치) + 관리자 한 명 더 승인(2-man rule).
- **Anomaly Detection**: 평소와 다른 시간대/지역/디바이스 로그인 시 webhook 알림 + 세션 즉시 만료.

### 6-2. 실시간 쓰기 감사 + 되돌리기

- 모든 mutation → Kafka/Redpanda 로그(또는 Supabase realtime) + 24시간 내 "되돌리기" UI.
- 에이전트 실수로 삭제한 매물 "휴지통 30일 보존" 표준 기능.

### 6-3. AI 안전 계층

- 모든 AI 응답(자연어 검색 파싱, 매물 설명 생성, 사진 분석)에 **출력 검증기** 경유: JSON schema + range clamp + profanity filter.
- OpenAI/Anthropic 호출 비용을 사용자 당 일일 한도로 제한 (초과 시 graceful 다운그레이드).
- AI 프롬프트 템플릿에 `system role` 격리 + user input 에 `<USER_INPUT>` 태그 마커 → 인젝션 탐지.

### 6-4. 프라이버시-First 데이터 레이어

- **필드 수준 암호화(FLE)**: 연락처/전화번호는 Supabase pgcrypto 로 application-layer 암호화. DB 덤프 유출 시에도 평문 노출 X.
- **PII minimization**: 게스트 검색 히스토리는 서버에 저장 X, 로그인 사용자만 저장 + 30일 auto-purge.
- **감사 로그 자체**의 암호화 + append-only(Supabase RLS `UPDATE FALSE, DELETE FALSE`).

### 6-5. 성능 차별화

- **Edge-first 쿼리**: 자주 나가는 viewport 쿼리 결과를 Cloudflare KV 에 5분 캐시. DB read 80% 감소.
- **이미지 파이프라인**: R2 → Cloudflare Images 자동 변환(AVIF + 다중 해상도). 모바일 3G 에서 지도 썸네일 500개 즉시.
- **Streaming SSR**: `/listings/[id]` 에서 OG 카드/핵심 정보 1차 스트리밍, 실거래가/유사매물은 Suspense 경계 분리.

### 6-6. UX 차별화

- **음성 자연어 검색**: 모바일에서 "강남구 아파트 5억 이하 방 2개" 음성 → Whisper → parseMatchQuery → 결과. 한 번도 타이핑 안 함.
- **매물 비교 AI 코멘트**: 선택한 매물 2~4개를 Claude 에게 넘겨 "전세 선택 시 고려할 점", "이 가격대에서 이 면적은 시세 대비 N%" 자동 생성.
- **통근 시뮬레이터 고도화**: 직장 위치 + 출근 시간대 → 교통 소요 시간 분포(지하철/버스/차) 시각화.

### 6-7. 개발자 경험

- **PR 자동 미리보기 배포** + Chromatic 시각 회귀.
- **보안 패치 정규화**: `L-sec*` 커밋은 반드시 (1) 테스트 추가, (2) 감사 로그 업데이트, (3) CHANGELOG.md 기재 3중 체크.

## 8. 결론

L-v7 과 L-sec100 시리즈로 WISHES 의 **코어 보안 표면**은 크게 개선됐다. 이번 감사는 두 가지 숨은 리스크를 찾아냈다: (1) **인가(Authorization) 계층이 인증(Authentication) 보다 한 단계 약하다** — 로그인은 되는데 "내가 이 리소스의 주인인가" 체크가 빠졌다. (2) **세션/자격증명 관리가 2020년 패턴** — localStorage/sessionStorage 에 민감값이 쌓여 있고 JWT 는 메모리에서 여러 계층으로 전달되고 있다.

2026년 기준 최선은: **세션은 HttpOnly cookie, 인가는 리소스 수준, 민감 작업은 패스키 + 2-man rule, 모든 mutation 은 감사 로그, AI 는 입출력 양방향 검증**. 위 로드맵대로 4주면 이 수준에 도달 가능하다.
