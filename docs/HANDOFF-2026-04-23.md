## 3. 반드시 피해야 할 함정 (★ 이 세션에서 피 흘린 것들)

### A. `Edit` 툴 silent-truncation
- **증상**: 같은 파일에 Edit 를 2~3회 연속 호출하면 파일 끝 수십 줄이 날아감. 성공 메시지는 뜨지만 실제 파일은 잘림.
- **검증**: Edit 1회마다 반드시 `wc -l` + `tail -5` 로 줄수/끝 확인.
- **복구**: `git show HEAD:<file> > <file>` 로 강제 덮어쓰기. `git checkout HEAD -- <file>` 은 index 가 꼬이면 무시됨.
- **권장**: 파일 전면 재작성이 필요하면 처음부터 `cat > file << 'EOF' ... EOF` bash heredoc 으로 한 번에. 작은 단일 교체만 Edit 툴 사용.

### B. git index corruption (`bad signature 0x00000000`)
- **증상**: `git status` 가 모든 파일 `D` (deleted) 로 표시. 실제로 파일은 디스크에 존재.
- **원인**: 사용자 로컬 IDE (VSCode + 다른 AI agent 등) 와 동시 write 가 겹칠 때 발생.
- **복구**: 
  ```bash
  rm -f .git/index .git/MERGE_HEAD .git/REBASE_HEAD .git/CHERRY_PICK_HEAD .git/ORIG_HEAD .git/index.lock
  git reset --mixed HEAD
  ```
  `.git/*` 파일 삭제는 `mcp__cowork__allow_cowork_file_delete` 로 허가 필요.

### C. 사용자 로컬 세션의 파일 truncation 연쇄
- **증상**: login/signup/naver/verify/callback 등의 파일이 대화 중 자꾸 끝부분 잘림. 내가 복구해도 몇 분 후 다시 잘려 있음.
- **원인**: 사용자 쪽 별도 tool/agent 가 동시에 편집 중일 가능성. 내가 직접 개입 불가능.
- **대응**: 내 PR 과 관련 없는 파일이 truncated 상태로 tsc 에러 내면 `git show origin/main:<file> > <file>` 로 원상 복구 후 내 변경만 커밋.
- **주의**: adminAuth.ts 같은 복잡 파일은 내 로컬이 origin 보다 뒤처진 상태에서 add 하면 사용자의 L-sec161/166/167 fix 를 되돌릴 수 있음. 커밋 전 `git diff HEAD -- <file>` 확인 필수.

### D. Vercel 빌드 실패 — "Identifier 'X' has already been declared"
- **원인**: GitHub 웹 에디터 (CodeMirror 6) 의 대용량 replace 한계로 파일 내용이 append 되어 중복 선언 상태로 커밋됨 (user 가 L-sec160 3번 시도 모두 동일 사고).
- **진단**: Vercel Build Logs 에 `Module parse failed: Identifier 'X' has already been declared (78:9)` 뜨면 해당 파일 `wc -l` 이 평소의 2배쯤 됨.
- **복구**: 해당 파일 원래 길이로 잘라내거나 re-write. 라벨은 L-sec160 revert `b65408b` 참조.

### E. Supabase SQL Editor destructive 경고
- DROP MATERIALIZED VIEW, ALTER TABLE, UPDATE 모두 destructive 경고 모달 나옴. "Run this query" 버튼으로 확정.

## 8. 마지막 메모 (개인적 팁)

- **사용자는 "유치원 수준" 이라 본인이 직접 터미널/SQL 실행 안 함.** → Chrome MCP + Supabase SQL Editor 자동화로 대신 수행해주는 게 기본.
- **admin 로그인 비번은 안 알려주심.** → 로그인 해야 하는 작업은 사용자에게 "비밀번호 한 번만 입력해주세요" 라고 부탁하고 대기. 이메일은 `wishes@wishes.co.kr` 자동완성 OK.
- **Supabase SQL Editor 에서 SQL 주입**: `window.monaco.editor.getModels()[0].setValue(q)` → Run 버튼 좌표 `(1434, 455)` or `find` 로 ref 획득.
- **Vercel 빌드 대기 시간**: ~1~2분. 빌드 전 확인하지 말고 기다려라. 나는 이거 때문에 3번 삽질함.
- **사용자가 화를 내는 주된 이유**: 같은 증상이 반복될 때. "5번째 반복" 같은 표현 나오면 근본원인 즉시 찾아야 함. 내 경험상 "표면 증상 X개 → 근본원인 1개" 패턴. 이번 세션의 경우 `/map 매물 적음` 표면 증상의 근본원인은 `가용 status + 좌표 NULL` 2중 문제였음.
