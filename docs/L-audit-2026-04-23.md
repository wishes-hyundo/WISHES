## 0. TL;DR

- **A-crit-1** — `/api/admin/contacts` PATCH 에 listing 소유권 체크가 없어 토큰만 있으면
  다른 중개사의 상담 status 를 임의 변경 가능했음 → **이번 커밋에서 수정 (L-sec136).**
- **A-crit-2** — `listings-bulk-delete` / `listings-bulk-update` / `listings-field-update`
  3개 엔드포인트가 `verifyAdminAuth` 만 통과시키고 `created_by` 체크가 없어 다른
  중개사의 매물을 대량 삭제/수정 가능했음 → **이번 커밋에서 수정 (L-sec136).**
- **A-obs-1** — `/search/page.tsx:24` IIFE 가 localStorage.admin_password 를
  sessionStorage 로 복제해서 레거시 풋건이 살아남아 있었음 → **이번 커밋에서 제거.**
- **L-sec112 재검증** — `/api/admin/listings/[id]` 는 이미 local authorizeListingMutation
  이 적용되어 있음 (커밋 82cce92 이후). 이번 커밋에서 본 로직을 `src/lib/adminAuthz.ts`
  로 추출해 4개 엔드포인트에서 공유하도록 일원화. [id]/route.ts 자체는 이 커밋에서
  손대지 않음 (blast radius 억제 — 다음 커밋에서 통합).

나머지 /map, /admin, /search 정밀 검수 결과는 섹션 3 이하 참조.

## 1. A-crit-1 : `/api/admin/contacts` PATCH IDOR

### 기존 상태 (vulnerable)
```ts
// verifyAdminAuth 통과 → 바로 update
const { data, error } = await supabase
  .from('contacts')
  .update({ status: parsed.data.status })
  .eq('id', parsed.data.id)
  .select()
  .single();
```
토큰이 어떤 agent 에게 발급된 것이든 `{id: <아무 contact.id>, status: '완료'}`
한 번의 PATCH 로 다른 중개사의 상담 상태를 임의 변경 가능. audit log 도 없음.

### 수정 (L-sec136)
1. `contacts.listing_id` 를 먼저 조회.
2. `authorizeListingMutation(request, listingId, supabase)` 로 해당 매물 소유자 체크.
3. `listing_id` 가 null 인 orphan 상담은 unlimited role (`master` / `superadmin` /
   `crawler_bridge`) 에게만 허용.
4. 모든 경로에 `audit()` 로깅: `.denied / .ok / .error`.

GET 핸들러는 read-only 여서 이번 phase 범위 밖. 추후 agent 필터(본인 매물 상담만
노출)는 UI 변경과 함께 별도 커밋.

## 2. A-crit-2 : 대량 매물 뮤테이션 3종 IDOR

### 공유 가드 추출 — `src/lib/adminAuthz.ts` (신규)
```ts
export async function authorizeListingMutation(request, id, supabase): AuthzSingleResult
export async function authorizeBulkListingMutation(request, ids, supabase): AuthzBulkResult
```
- `UNLIMITED_ROLES = { master, superadmin, crawler_bridge }` → 소유자 체크 생략 (운영 필요).
- agent role → `uid === listings.created_by` 만 통과.
- `created_by` null 인 레거시 매물은 unlimited role 만 건드릴 수 있음.
- Bulk 는 `ownedIds` (소유 인덱스) 와 `filteredOut` (차단된 id) 을 분리 반환 →
  호출측은 **반드시 `ownedIds` 를 `.in()` 필터로 사용해야** IDOR 재발 방지.

### `listings-bulk-delete` (before → after)
| 경로 | Before | After |
| --- | --- | --- |
| `ids[]` | 토큰만 검증, ids 전부 삭제 | `authz.ownedIds` 로만 삭제 · 소유 아닌 id 는 응답에 `skipped: <count>` (id 리스트 미노출) |
| `source_site` | 토큰만 검증, 해당 source 전체 삭제 | unlimited role 일 때만 허용 (agent 403) |
| audit | 없음 | `.denied / .ok / .error` 모두 기록 |
| rate limit | 1h 10회 (L-sec124) | 유지 |

### `listings-bulk-update` (before → after)
- 기존: `createClient(url, SERVICE_ROLE_KEY)` 직접 사용 + 소유 체크 없음.
- 수정: `createServerClient()` 사용 (최소권한 원칙, service_role 직접 주입 제거).
  `authorizeBulkListingMutation` 으로 `ownedSet` 구성 후 per-item 루프에서
  `ownedSet.has(item.id)` 아니면 `{ success: false, skipped: true, error: 'forbidden' }`.
- 응답에 `totalSkipped` 추가, audit meta 에 `{requestedCount, ownedCount, filteredOutCount, updated, skipped, failed, bypassed}`.

### `listings-field-update` (before → after)
- PUT(단건): `authorizeListingMutation` + audit.
- POST(벌크): `authorizeBulkListingMutation` + per-item ownedSet 체크.
- **신규 rate limit** on POST: `field-update-bulk:ip:${ip}` 30분 40회 (기존 누락).
- `id` 타입 가드 `typeof id === 'number' && Number.isInteger(id)` 로 강화.

### 공통 설계 결정
1. **IDS 를 그대로 DB 에 넘기지 않는다.** 소유 필터 결과만 사용. 이게 L-sec136 의 회귀 락.
2. **Skipped id 리스트는 응답에 노출하지 않음** (info leak 방지). 개수만 반환.
3. `UNLIMITED_ROLES` 집합은 한 곳에 정의 (adminAuthz.ts). 후속 엔드포인트 추가 시
   복붙 대신 import.
4. `/api/admin/listings/[id]` 의 local authorizeListingMutation 은 이번 커밋에서
   손대지 않음. 리팩터 통합은 다음 커밋 (블래스트 반경 억제).

## 3. A-obs-1 : `/search` 레거시 localStorage.admin_password 복제

### 기존 상태 (`src/app/search/page.tsx:24`)
```ts
// IIFE 내부: ws_token / ws_user / ws_login_time 외에...
var p=localStorage.getItem('admin_password');
if(p) sessionStorage.setItem('admin_password', p);
```
`admin_password` 는 L-sec113 이후 서버측 의존이 사라진 레거시 fallback 자격증명.
이 IIFE 한 줄이 새 탭을 열 때마다 localStorage 값을 sessionStorage 로 자동 복제해
사실상 legacy 를 유지하고 있었음. C-2 phase 2/3 진행과 무관하게 먼저 제거 가능.

### 수정 (L-sec136)
해당 한 줄만 삭제. `ws_token` 마이그레이션 경로는 유지 (정상 로그인 세션 복원).
주석으로 L-sec113/C-2 문맥 명시.
